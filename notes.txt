Section 2: Foundations: LangChain, Agents & Flow
    6. Modern AI Apps: UI -> Orchestration -> Models & Tools 
        - Modern AI Apps = 3 Layers
        - UI -> Orchestration -> Models & Tools
            - ui: user interface like chat, cli, etc
            - Orchestration: brain + wiring of your AI application, it will decide that which model
            going to call, which tools or APIs it will use, whether we need a search or whether we need a rag
            or we need a databases or etc. and combining the multiple steps into a particular proper flow.
            - Models & Tools: models can be openAI or Gemini or local models. tools can be web search, it can be 
            database queries, internal APIs, vector DB, calculators, it can be any tools.
        - Think beyond "ChatGPT Wrapper"
        - Orchestration = brain & wiring

    7. What is an AI Agent
        - Agent = LLM + memory + tools + rules
            - An Agent is an LLM powered program that can use tools that can look at any context,
            then based on the context, it can decide the next step and that decision on the next step 
            that will be based on the rules or the goals that you are going to set
        - Can decide "What to do next"
        - Still under your control

        diagram:
            user request -> agent logic: check context, call tools, draft answer, maybe repeat, ask user approvals -> final response

            real example (support agent):
                msg -> KB (knowledge base) -> draft a reply with citations(sources can be web browser or llm) human approvals

    8. Chains vs Agents
        - Chains = fixed steps (no decision)
            - fixed pipeline, all the steps are defined
            - user input -> system prompt -> call model -> answer
            - eg(rag): retrive docs -> prompt+docs -> call model -> answer
        - Agent = Chooses next Step using context
        - Use chain first, agents when needed

    9. Where LangChain Fits in this Stack
        - LangChain = toolkit for LLM Workflows, and provide consistent wrappers over different models (switch to multiple model seamlessly)
        - prompts, models, tools, RAG, LCEL
        - Used inside the Orchestration layer

        digram:
            UI -> Orchestration with LangChain -(communicate)> models, tools, vector dbs

    10. Where LangGraph Fits in this Stack
        - LangGraph = graphs for agent Workflows
        - Nodes, edges, state, retries, HITL
        - Sits above LangChain building blocks

        LangChain -> build steps
        LangGraph -> how those steps run over time

        - LangChain is the tool or concept which will talk to the models
        - LangGraph is going to Orchestrate multi-step agent like flows

        diagram:
            UI -> LangGraph (define the graph, state, control flow) -> Call LangChain -> models, Db, Apis

    11. JSON-first Mindset
        - Treat LLM Output as data, not paragraph
        - JSON = predictable, testable, integratable
        - We enforce shapes with zod

Section 4: LLM Fundamentals: JSON-First Approach
    20. Tokens, const and context window
        - Tokens = chunk of words
        - Context Window = how mush model can see 
            - system instructions, user input, chat history, tools responses, its own output
            - beyond the limit possibly the older part get truncated or dropped
            - rules:
                - prompt should be very lean
                - have a control in amount of context should be sent

                - more tokens = more to read for llm
                - more tokens out = more to generate 
                - cost and latency increases

                - short and concise > long, random dumping
        - Too Long -> truncation, higher cost

    21. Sampling Knob: Temperature, Top_p, Max_tokens
        - temperature: stability vs creativity
            - 0 - 1
            - 0 - 0.3 => stable temperature, deterministic, great for tools and json
            - more that 0.7 => more creative, good for brainstorming
        - top_p: how wide to Sampling
            - how much of the probability must to consider  
            - temperature low for reliablility or tupe top_p for more or less variety
        - max_tokens: cap response length
            - used to protect cost
            - used to keep responses concise
        - penalties: reduce repetition

        tempreature
            - 0 -> safest/short answer (best for JSON)
            - 0.3 - 0.7 -> more variety; can drift the JSON
            - default for JSON: 0
        top_p
            - 1 -> consider all likely words (simple default)
            - 0.9 -> ignore the long tail (better for JSON)
            - beginner rule: tune either temperature or top_p, not both
        max_tokens:
            - Hard cap on reply size
            - Too small -> cut-off JSON
            - for (summary, confidence): start at 256

        important: High tempreature or a tiny max_tokens are the two most common causes of malformated JSON 

        Provider Naming - Same Knob, Different labels
        while parameter names vary by provider, the underlying concepts remains consistent

        parameter:
            temperature:
                - openAI => temperature
                - gemini => generationConfig.temperature
                - ollama => options.temperature
            top_p:
                - openAI => top_p
                - gemini => generationConfig.topP
                - ollama => options.top_p
            Max Tokens:
                - openAI => max_tokens
                - gemini => generationConfig.maxOutputTokens
                - ollama => options.num_predict
        we adapt names per provider under the hood; you'll think in the same three knob everywhere

    22. Chat models vs Tools
        - Chat models: free-form answers
            - best for conversation, explanation, general Q/A
            - follow instruction but sometime it not going to give strict format answer back
        - Tool/JSON models: better for functions & schemas
            - tools decide to follow your schema
            - call the tools and based on that create valid json or structured json properly in same defined format
        - Choose based on task, not hype
            - Chat vs Tools — Which Model When?

            Chat Models:
            You send messages (system + user), it replies with text.

            // Chat Model

            messages: [system, user]
            response: text

                Choose Chat when:
                    Pure language tasks (summary/explain/rephrase)
                    Compact JSON answer ({ summary, confidence })
                    No external APIs/browsing/files needed

            Tools Models

            Model can decide to call a function you define with a name + JSON schema.

            // Tools Model

            functions: [weather, websearch, db]

            response: text + function calls
           
            Choose Tools when:
                Outside information needed (search, fetch URL)
                Model chooses tools and order of execution
                Plan → tool → observe → decide loop is acceptable

    23. Zod schemas as contracts for outputs
        - zod = define expected shapes
        - Same Schema -> runtime + Typescript
        - Trust, but verify model output

        Zod Schema — Contract for Typed JSON
        Why Zod?
            - Describes the shape of data at runtime
            - Guarantees CLI returns { summary, confidence } in the right shape
            - With withStructuredOutput(...) guides model + auto-parses/validates
        Key Benefits
        // Zod ensures your JSON is always valid
        {
        summary: string,
        confidence: number
        }

        Installation + Authoring
        Install
        pnpm add zod
        Create the schema
        // packages/shared/src/schema.ts

        import { z } from "zod"

        export const JsonResponseSchema = z.object({
        summary: z.string(), // A short, plain-English answer

        confidence: z.number() // 0.0 to 1.0
        })

        export type JsonResponse = z.infer<typeof JsonResponseSchema>

        This Zod schema is the contract between the model and our app.

    24. Structured Output
        - Ask model to fill schema, not guess format
        - provider features help: JSON mode, tool caling
        - More reliable that "pls respond in JSON"

        - if not structured output we will receive wrong data type in the json.

        schema -> structured output mode -> predictable json

Section 6: LangChain fundamentals
    33. Why use LangChain JS over raw SDKs
        - Raw SDKs = low-level calls
        - LangChain = shared patterns & utilities
        - Easier to switch models & compose

        purpose: (easy integration)
            - multiple model switching
            - tools
            - RAG
            - retries
            - logging

        LangChain? AI toolkit work with multiple model providers, simple abstraction for prompts, output, 
        parsing tools, retrievers, working with vector db

        LCEL -> pipeline to compose multiple functions together

    34. Core building blocks
        - Prompt: what to ask + how
            - 1. system role
            - 2. instructions
            - 3. placeholder for variables

            - prompt should be predictable, reusable and earier to read
        - Model: which LLM we call

        - Output Parser: turns text -> data

    35. Runnables & LCEL
        - Runnables: Generic unit of work that can be invoked, batched, streamed
            - each step in LLm pipeline is runnable
            - runnables -> takes input -> does something -> returns output
            runnables are (prompt template, calling model, getting result)

            - runnable sequence (run steps in order), 
            Where the output of each is the input of the next (a->b->c->d->e)
            - runnable map (run multiple runnables in branches)
        - LCEL: LangChain Expression Language
            - it is used to connect those runnables together
            - .pipe, .invoke
